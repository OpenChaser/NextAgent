## 一、项目概览

NextAgent 是一个基于 **Electron + React** 构建的跨平台 AI 工作助手桌面应用，目标是集成 AI 对话、内置工具调用、任务/项目/自动化等能力于一体。

### 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面框架 | Electron 29 | 跨平台桌面应用容器 |
| 渲染层 | React 18 + TypeScript 5 | UI 与类型安全 |
| 构建工具 | Vite 5 | 渲染进程极速构建/HMR |
| 样式 | Tailwind CSS 3 | 原子化 CSS |
| 图标 | Lucide React | 轻量图标库 |
| AI SDK | openai 4.x | 兼容 OpenAI 协议的多家模型接入 |
| 包管理 | pnpm 9 | monorepo 友好、磁盘节省 |
| 打包 | electron-builder | 产出 nsis 安装包 |

### 进程架构

```
┌─────────────────────────────────────────────┐
│ Main Process (src/main)                      │
│  窗口管理 / 菜单 / IPC / 模型配置 / AI 调度  │
└──────────────┬──────────────────────────────┘
               │ contextBridge (preload)
┌──────────────┴──────────────────────────────┐
│ Preload (src/preload/preload.ts)            │
│  安全暴露 electronAPI 给渲染进程             │
└──────────────┬──────────────────────────────┘
               │ window.electronAPI
┌──────────────┴──────────────────────────────┐
│ Renderer (src/renderer)                      │
│  React UI: Sidebar / ChatArea / Popover      │
└─────────────────────────────────────────────┘
```

核心安全策略：`nodeIntegration: false` + `contextIsolation: true`，渲染进程不直接接触 Node 能力，全部经 preload 桥接，符合 Electron 安全最佳实践。

## 三、关键技术点与经验

### 1. 主进程流式聊天 + 工具调用循环

`src/main/main.ts` 中的 `chat:send` handler 是整个 AI 能力的核心。经验要点：

**流式分片累积 tool_calls**
OpenAI 流式返回时，`tool_calls` 是按 `index` 分片到达的，不能直接用最后一个 chunk。当前实现用 `Map<index, buffer>` 累积 `id` / `name` / `arguments`，这是处理流式工具调用的正确姿势：

```ts
const toolCallsBuffer = new Map<number, { id; name; arguments }>()
for (const tc of delta.tool_calls) {
  const idx = tc.index
  if (!toolCallsBuffer.has(idx)) toolCallsBuffer.set(idx, { id: tc.id || '', name: '', arguments: '' })
  const buf = toolCallsBuffer.get(idx)!
  if (tc.function?.name) buf.name = tc.function.name
  if (tc.function?.arguments) buf.arguments += tc.function.arguments
}
```

**多轮工具循环**
用一个 `MAX_ROUNDS = 10` 的 for 循环驱动「模型生成 → 若有 tool_calls 则执行并回填 → 继续」的 agentic 循环，直到模型不再产出工具调用或达到轮次上限。这是 Agent 的最小可用骨架。

**经验教训**
- 一定要限制最大轮次，否则模型若陷入「不断调用工具」的循环会把进程跑满。
- `stream_options: { include_usage: true }` 才能在流末尾拿到 token 用量。
- 每次 `chat:chunk` / `chat:tool_call` / `chat:done` / `chat:error` 都通过 `win.send` 推给渲染进程，前端按事件渲染即可。

### 2. 模型配置：默认文件 + 用户目录覆盖

模型配置文件存放在 `~/.nextagent/models.json`，首次启动时若不存在则从内置默认文件 `src/data/model.json` 拷贝一份（打包后从 `dist/data/` 读取）。

**经验要点**
- 用户态配置放 `os.homedir()` 下，避免应用升级覆盖用户密钥。
- 打包/未打包路径不同：`app.isPackaged` 时默认文件位于 `__dirname/../data`，开发态位于 `__dirname/../../src/data`，需分别处理。
- **安全提醒**：当前 `key` 以明文存于本地 JSON，仅适合个人工具级别。若要分发，应引入系统 keychain（如 `keytar`）或加密存储。

### 3. 工具系统设计

`src/main/tools/` 下每个工具都是一个 `ToolDefinition`，统一在 [index.ts](src/main/tools/index.ts) 注册：

```ts
export const allTools: ToolDefinition[] = [ readFileTool, writeFileTool, ... ]
```

`getToolDefinitions()` 返回给 SDK 的纯 schema（不含 executor），`executeTool()` 执行具体调用。**声明与执行分离**的好处是 schema 可序列化给模型、executor 留在主进程内部，边界清晰。

工具内部要注意：
- `read_file` 对返回长度做了 `50000` 字符截断，避免一次性塞爆上下文——这是个真实踩过的坑，值得保留。
- 工具执行出错时返回 `错误：...` 字符串而非抛异常，保证 Agent 循环不被中断。

### 4. Preload 桥接与监听器清理

[preload.ts](src/preload/preload.ts) 用 `contextBridge.exposeInMainWorld('electronAPI', ...)` 暴露受控 API。其中特别提供了 `removeChatListeners()`，用于在组件卸载或重发消息前清掉旧的 `chat:*` 监听器。

**经验教训**：这是修复「重复回复窗口」问题（提交 `d4f3257`/`e92b8ae`）的关键——IPC 监听器若不清理，多次发送会叠加多个 listener，导致同一条回复被渲染多次。React 组件里务必在重发前调用 `removeChatListeners()`，并在 `useEffect` cleanup 里同样清理。

### 5. 构建/类型检查的「双进程」问题

Electron 有主进程（Node 环境）和渲染进程（浏览器/DOM 环境），两者 TS 配置不同：

- `tsconfig.main.json`：主进程，含 Node 类型
- `tsconfig.json`：渲染进程，含 DOM/React 类型

Vite 用 esbuild 转译渲染进程时**不做类型校验**，所以 `pnpm run typecheck` 单独跑两份 `tsc --noEmit`，是 CI 门禁的第一道关卡，弥补了 Vite 的缺口。这一点写进了 [AGENTS.md](AGENTS.md) 与 [README.md](README.md)，每次上库前必须 `typecheck` + `build` 双跑。

## 四、踩坑记录

### 4.1 pnpm-workspace.yaml 缺少 packages 字段

**现象**：`pnpm install` 报错，提示 workspace 配置异常。

**根因**：`pnpm-workspace.yaml` 里没有 `packages` 字段，pnpm 9 严格要求该字段存在（即使非 monorepo）。

**解决**：添加 `packages` 字段（提交 `70d602d` / `1beea2f`，PR #4 #5）。

**经验**：pnpm 9 的破坏性变更，升级包管理器版本时务必关注 release note。

### 4.2 Electron 二进制未下载

**现象**：启动时报 `Electron failed to install correctly`，`node_modules/electron/dist/electron.exe` 缺失。

**根因**：`pnpm-workspace.yaml` 设置 `allowBuilds: electron: false` 跳过了二进制下载（节省 CI/不必要安装）。

**解决**：用阿里云镜像补装：

```powershell
$env:ELECTRON_MIRROR='https://npmmirror.com/mirrors/electron/'
node node_modules/electron/install.js
```

**经验**：国内环境下务必配镜像；该命令已固化进 [AGENTS.md](AGENTS.md) 的「依赖安装」段，新人无需重复踩坑。

### 4.3 流式 tool_calls 分片丢失

**现象**：偶发工具参数 JSON 解析失败。

**根因**：早期实现取了流式最后一个 chunk 的 `tool_calls`，而参数是分片到达的，导致 `arguments` 不完整。

**解决**：改用 `Map<index, buffer>` 累积（见上文 3.1）。同时 `JSON.parse` 失败时降级为空对象 `{}`，保证循环不崩。

### 4.4 端口 5173 被占用

**现象**：`pnpm start` 时 Vite 报 `Port 5173 is already in use`。

**根因**：上次 dev server 进程未正常退出。

**解决**：PowerShell `Get-NetTCPConnection -LocalPort 5173` 找进程并 kill（已写进 README 故障排除段）。

## 五、工程化实践

### 5.1 CI 门禁

[.github/workflows/build.yml](.github/workflows/build.yml) 在 push/PR 到 main 时执行：
1. `pnpm install --frozen-lockfile`
2. `pnpm run typecheck`
3. `pnpm run build`

并配置了 `concurrency` 自动取消同分支旧运行，节省 CI 资源。本地上库前用同样两条命令自检，避免门禁红灯。

### 5.2 Git Worktree 多特性并行

项目用 git worktree 并行开发多个特性，约定如下（已在本仓库落地）：

| Worktree 路径 | 分支 |
|---------------|------|
| `D:/github/NextAgent` | `dev`（主开发） |
| `D:/github/NextAgent-feature-dev-worktree` | `feature/dev-worktree` |
| `D:/github/NextAgent-feature-list` | `feature/feature-list` |
| `D:/github/NextAgent-feature-task-list` | `feature/task-list` |
| `D:/github/NextAgent-feature-dev-experience-summary` | `feature/dev-experience-summary` |

**经验**：worktree 让每个特性一个独立目录，切换无需 stash，编译产物互不干扰；路径命名统一 `NextAgent-feature-<英文特性名>`，分支统一 `feature/<英文-kebab-case>`。

### 5.3 国内镜像加速

统一使用阿里云镜像，写进 [AGENTS.md](AGENTS.md)：

- npm registry：`https://registry.npmmirror.com`
- Electron 二进制：`https://npmmirror.com/mirrors/electron/`

安装依赖：`pnpm install --registry=https://registry.npmmirror.com`。

## 六、后续改进建议

以下是当前阶段观察到的可改进点，供后续迭代参考：

1. **密钥安全**：模型 API Key 明文存储，建议接入 `keytar`（系统 keychain）或加密存储，避免分发场景泄露。
2. **工具权限**：内置工具（如 `run_command`、`write_file`）直接执行，缺少用户确认环节。建议引入「工具调用确认」机制（参考 [PermissionPopover.tsx](src/renderer/components/PermissionPopover.tsx) 的雏形），危险工具执行前弹窗确认。
3. **会话持久化**：当前聊天历史仅在单次请求的 messages 数组里，未落盘。建议持久化到 `~/.nextagent/sessions/`，支持多会话切换。
4. **错误处理标准化**：工具执行错误目前返回中文字符串，模型需自行理解。可考虑返回结构化 `{ ok: boolean, error?: string, data?: string }` 并在 schema 里约定。
5. **Token 预算**：已有 `max_input_tokens`，但未在多轮循环中做主动截断。建议接近上限时自动摘要历史，避免请求被模型拒收。
6. **测试**：目前无单测/集成测试，建议至少对 `tools/index.ts` 的执行器与 `main.ts` 的流式累积逻辑补测，CI 增加测试步骤。
7. **类型复用**：`Model` / `ModelConfig` 在 main 与 preload 各定义了一份，建议抽到共享 `src/shared/types.ts`，单点维护。
